# LBank API 制限事象レポート — IZKY/USDT 取引履歴取得不可

報告日: 2026-03-04

---

## 重要度: Phase1 ブロッカー

本事象は単なる機能欠損ではなく、**Phase1 の成功定義を満たせないブロッカー**です。

### Phase1 成功定義（仕様書 1.2）

> - 1取引所 × 1ペア × 1Cup が正常開催される
> - **実データでランキングが生成される**
> - 管理者が結果確定できる

「実データでランキングが生成される」は、**IZKY/USDT の取引実績に基づく正確な PNL** が前提です。
取引履歴 API が使えない現状では、この前提が崩れます。

---

## 事象の概要

LBank の認証付き API で **IZKY/USDT ペアの取引履歴が取得できない**ことが判明しました。

---

## 影響範囲

### 1. 出来高条件が機能しない

| 項目 | 影響 |
|---|---|
| 出来高（Volume）の取得 | 不可（全参加者が $0 と表示される） |
| 最低出来高条件の判定 | 不可（全参加者が「対象外」になる） |
| 関連仕様 | 仕様書 3.2④「最低 100 USDT 相当の総出来高を達成した場合のみランキング対象」 |

### 2. PNL の正確性が担保できない

残高取得 API（USDT残高・IZKY残高）は正常に動作していますが、**口座全体の残高**を取得しているため、以下のリスクがあります。

**問題: 他ペアの取引損益が PNL に混入する**

例）参加者が BTC/USDT を取引して USDT 残高が変動した場合：

| 時点 | USDT | IZKY | 総資産 |
|---|---|---|---|
| 開始 | 100 | 10,000 | 112.00 |
| 終了（BTC売却で+50 USDT） | 150 | 10,000 | 162.00 |

→ IZKY を一切取引していないのに **PNL +44.6%** と表示される

取引履歴がないため：
- USDT 残高の変動が IZKY 取引によるものか判別できない
- 他ペアの取引損益を除外できない
- 出来高条件でフィルタすることもできない

**→ 「IZKY/USDT の実データに基づくランキング」とは言えない状態になる**

---

## 技術的な詳細

### 検証したエンドポイント

| エンドポイント | 用途 | IZKY/USDT での結果 |
|---|---|---|
| `POST /v2/supplement/transaction_history.do` | 約定履歴の取得 | **error 10008: currency pair nonsupport** |
| `POST /v2/spot/trade/orders_info_history.do` | 注文履歴の取得 | **error 10008: currency pair nonsupport** |
| `POST /v2/supplement/user_info_account.do` | 残高の取得 | 正常 |
| `GET /v2/ticker/24hr.do` | IZKY 価格の取得 | 正常 |
| `GET /v2/currencyPairs.do` | ペア一覧の取得 | `izky_usdt` が含まれている |

### 検証条件

- APIキーを2回再発行して検証（IP制限の問題ではないことを確認）
- LBank の公開ペア一覧には `izky_usdt` が含まれており、取引自体は可能
- 同様の事象は CCXT（暗号資産取引所ライブラリ）でも報告されている
  - https://github.com/ccxt/ccxt/issues/18970
  - LBank サポートの回答:「ほとんどの取引ペアは取引履歴 API を公開していない」

### 結論

**LBank 側の仕様制限であり、実装側での回避策はありません。**

---

## 対応方針

### A. 出来高条件の一時無効化（暫定対応）

全参加者をランキング対象とし、出来高条件を一時的に無効化する。

- メリット: PNL ランキング自体は即座に動かせる
- デメリット: 他ペアの損益混入リスクが残る。Phase1 成功定義を厳密には満たさない
- 適用場面: リリースを優先し、参加者に「IZKY/USDT 以外の取引を控えてください」と運用で対処する場合

### B. LBank への対応依頼

LBank サポートに以下を依頼する：

1. IZKY/USDT ペアの `transaction_history.do` / `orders_info_history.do` の対応開放
2. 対応が不可能な場合、代替手段（Webhook 等）の提案

- メリット: 既存実装をそのまま活用できる
- デメリット: 対応時期が LBank 側に依存（不確定）

### C. IZKY/USDT を扱う別の取引所に切り替える

取引履歴 API が IZKY ペアに対応している取引所に変更する。

- メリット: LBank 側の対応を待たずに Phase1 成功定義を満たせる
- デメリット: API 実装の差し替えコスト（署名方式・エンドポイント・レスポンス形式）
- 前提: その取引所の API で取引履歴が IZKY ペアに対応していること（事前検証必須）

---

## クライアントへの確認事項

以下の情報をいただけると、対応方針の判断が早まります。

1. **IZKY が上場している他の取引所の一覧**
   - 取引履歴 API の対応可否を事前検証します
2. **ユーザーが主に利用している取引所はどこか**
   - 取引所変更時の参加者への影響を評価します
3. **LBank サポートへの問い合わせは可能か**
   - API 対応開放の依頼を並行で進められるか
